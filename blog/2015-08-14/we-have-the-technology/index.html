Free Code Fridays: We Have The Technology

Daniel Blakemore • Aug 14, 2015

Last week I hinted at a Cocoapod featuring a camera and photo picker. That particular codebase is cruftier than I had originally remembered. In addition, it doesn't have a meaningful example project showcasing how easy it is to use. While we prepare it for prime time, I'd like to highlight some wonderful work by our own Kevin Wong and a pod I forgot to mention in last week's post.

PXViewController: Kevin Wong saves the day!

Last week I talked about the additions we had made to several iOS view controllers. The way that pod came together was one view controller at a time and in the README.md for the pod I mentioned that I hadn't figured out a way to merge all the code nicely so that all subclasses of the various view controllers used a common version of each function. The benefits of not having four copies of each function are obvious as that kind of duplication is a classic example of a missed opportunity for code reuse.

At the time of writing that, Spencer Phippen and I were devising an NSProxy-based approach that would allow us to forward all the methods from the various view controllers to a common object that would handle the functionality. At the end of the week we had a semi-functional but inelegant test workspace that didn't quite handle UI_APPERANCE_SELECTORs right. Our plan was to fix those issues, clean up the code, and integrate the changes into the PXViewController pod. Our efforts would have been in vain.

Over the weekend Kevin took it upon himself to take the published pod and entirely refactor it based on a category and lots of swizzling. There was a small code review and a few fixes for fiddly functionality applied, but Kevin rebuilt my cluttered, slightly hacky subclasses into a state-of-the-art swizzling machine. Maintenance, bug fixes, and new features will now be 300% easier to implement all thanks to the weekend whims of our resident super engineer.

Thanks Mr. Wong!

UIImageView_AFNetworking-Blocks: Slipping through the cracks

When I set out to write this series, I looked at all of the private CocoaPods we had and sorted them based on dependency so I could publish things that built on top of the previous posts. I then grouped them all into manageable chunks and saved my list, never looking back. It turns out that I made (at least one) small mistake on the list. Our "category" on AFNetworking's UIImageView category had not been included, even though PXImageView from last week depended on it. It’s on our GitHub now, but this will be the first writeup about it.

If you haven't noticed from the code released so far, I really like blocks. One wonderful thing about AFNetworking is the block-based interfaceit provides for dealing with networking request responses. Unfortunately, one of the very useful categories that comes with it, UIImageView+AFNetworking doesn’t have the same format. You could tell an image view to load its image from a URL, but you had no idea when it was done (Note: there is a method with completion blocks for setting an image with an NSURLRequest, but not a corresponding method that takes an NSURL). I made a "category" that lets you specify a completion to be called when the loading finishes (or fails). The reason I say "category" is because it is really a category on UIImageView that contains the same interface as the AFNetworking category, but performs a load that allows a callback to be called when the image is finished. This pod should be able to drop straight into your project and provide the additional methods with the completions.
