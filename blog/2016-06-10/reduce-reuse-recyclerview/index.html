Let's Code: Reduce, Reuse, RecyclerView

Calvin Kern • Jun 10, 2016

If you’re still using ListView in Android, boy have I got a treat for you! Today, I wanted to talk about one of my favorite UI components in the Android SDK, RecyclerView, which is basically a ListView. RecyclerView is used to display rows (or grids) of items to the end user from some dataset. Built in to the RecyclerView are animations for adding, deleting, and moving items in the list, so we get all of that material design goodness with some simple API calls.

The main difference between a RecyclerView and a List View is the use of the view holder design pattern. This pattern makes use of view holders which bind their data into 'recyclable' views. A view becomes 'recycled' when it is scrolled off screen and will try to be used again by another view holder such that only the views that ever need to be on screen are created. This helps to improve the efficiency of your application, especially when scrolling in the RecyclerView. A view holder class also holds onto references for the views that it will bind data to later. This is a performance boost the RecyclerView offers as you won't need to find the view by id every time you want to update the UI for that view holder.

In this silly little demo, let's build an app that has a header, a place for a contact image, and then a bunch of information about said contact. We will also be able to add items to the adapter and remove them by receiving clicks in a view holder!

Let's start with a new project in Android Studio with just a blank activity and xml layout file. There is some setup to get a RecyclerView in and working, but it won't take us very long. First, let's fill out the XML file with a simple layout containing our RecyclerView:

<RelativeLayout
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    >
    <android.support.v7.widget.RecyclerView
        android:id="@+id/recycler"
        android:layout_width="match_parent"
        android:layout_height="match_parent"
        />
</RelativeLayout>

Then in the activity, we need to set the layout manager and the adapter of our RecyclerView. Our app doesn't require much, so we can use a simple LinearLayoutManager to display our rows in a linear fashion. When creating the layout manager, you can specify whether it should lay out vertically or horizontally, as well as if the order should be reversed (start at the bottom versus the top). Let’s create a bare bones adapter for now with a simple list that maintains all the data to be displayed.

public class MainActivity extends AppCompatActivity {

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        // Get the recycler view and set up the layout manager as well as the adapter
        RecyclerView recycler = (RecyclerView) findViewById(R.id.recycler);

        if (recycler != null) {
            recycler.setLayoutManager(new LinearLayoutManager(MainActivity.this));
            recycler.setAdapter(new MainRecyclerAdapter());
        }
    }

    public class MainRecyclerAdapter extends RecyclerView.Adapter {
        List<String> mInfoStrings;

        public MainRecyclerAdapter() {
            mInfoStrings = new ArrayList<>();
        }

        @Override
        public RecyclerView.ViewHolder onCreateViewHolder(ViewGroup parent, int viewType) {
            return null;
        }

        @Override
        public void onBindViewHolder(RecyclerView.ViewHolder holder, int position) {
        }

        @Override
        public int getItemCount() {
            return 2 + mInfoStrings.size(); // 2 = Header + Contact
        }
    }
}

Next, let's define some view types in MainRecyclerAdapter to use in our RecyclerView. We will need a view type for the header holder, the contact holder, and the info holder. 

private static final int HEADER_TYPE = 0;
private static final int CONTACT_TYPE = 1;
private static final int INFO_TYPE = 2;

In order to really make use of these view types, we also need to override the getItemViewType() method in our adapter. This method will be used to determine which view type is at some position, and can help us when binding data to our view holders a little later on.

@Override
public int getItemViewType(int position) {
    if (position == 0) {
        return HEADER_TYPE;
    } else if (position == 1) {
        return CONTACT_TYPE;
    } else {
        return INFO_TYPE;
    }
}

Back in MainActivity.java, lets define some classes for our view holders. 

public class MainActivity extends AppCompatActivity {
    //...
    public class HeaderViewHolder extends RecyclerView.ViewHolder {

        public TextView mHeaderTextView;

        public HeaderViewHolder(View itemView) {
            super(itemView);

            mHeaderTextView = (TextView) itemView.findViewById(R.id.holder_header_text_view);
        }
    }

    public class ContactViewHolder extends RecyclerView.ViewHolder {

        public TextView mContactNameView;
        public ImageView mContactImageView;

        public ContactViewHolder(View itemView) {
            super(itemView);

            mContactNameView = (TextView) itemView.findViewById(R.id.holder_contact_name_view);
            mContactImageView = (ImageView) itemView.findViewById(R.id.holder_contact_image_view);
        }
    }

    public class InfoViewHolder extends RecyclerView.ViewHolder {

        public View mDeleteView;
        public TextView mInfoTextView;

        public InfoViewHolder(View itemView) {
            super(itemView);

            mDeleteView = itemView.findViewById(R.id.holder_info_delete);
            mInfoTextView = (TextView) itemView.findViewById(R.id.holder_info_text_view);
        }
    }
}

One last step to finish the setup of these view holders, we need to actually create them in our adapter's onCreateViewHolder()! This will be a cinch since we've already set up our view types.

@Override
public RecyclerView.ViewHolder onCreateViewHolder(ViewGroup parent, int viewType) {
    LayoutInflater inflater = LayoutInflater.from(parent.getContext());

    RecyclerView.ViewHolder holder; // Declare the ViewHolder to initialize later
    switch (viewType) {
        case HEADER_TYPE: {
            holder = new HeaderViewHolder(inflater.inflate(R.layout.holder_header, parent, false));
            break;
        }
        case CONTACT_TYPE: {
            holder = new ContactViewHolder(inflater.inflate(R.layout.holder_contact, parent, false));
            break;
        }
        case INFO_TYPE:
            // Intentional overflow
        default: {
            holder = new InfoViewHolder(inflater.inflate(R.layout.holder_info, parent, false));
            break;
        }
    }
    return holder;
}

Now we're finally ready to fill in the method onBindViewHolder() in our MainRecyclerAdapter. This method is responsible for updating the view holder’s UI components will the data that we need. By having public variables in our view holders, our adapter can directly control what those values display. Let's go ahead and fill in some simple data so we can actually see something when we build the app. Notice the use of getItemViewType() here which we wrote earlier to help us determine which view holder type we are trying to bind data to.

@Override
public void onBindViewHolder(RecyclerView.ViewHolder holder, int position) {
    switch (getItemViewType(position)) {
        case HEADER_TYPE: {
            ((HeaderViewHolder) holder).mHeaderTextView.setText("This is a title!");
            break;
        }
        case CONTACT_TYPE: {
            ((ContactViewHolder) holder).mContactNameView.setText("Person Name");
            ((ContactViewHolder) holder).mContactImageView.setImageResource(R.drawable.background);
            break;
        }
        case INFO_TYPE: {
    ((InfoViewHolder) holder).mInfoTextView.setText(mInfoStrings.get(position - 2));
            break;
        }
    }
}

All right, now that we have a basic working RecyclerView! Before we go any further, let’s add a method to our MainRecyclerAdapter that lets us insert strings to the dataset. We can make use of the built in insertion animations by using notifyItemInserted() rather than just refreshing the whole list with notifyDataSetChanged().

public void addInfoString(String info) {
    mInfoStrings.add(info);
    // Notify an insert of the item into the end of the info section
    notifyItemInserted(getItemCount() - 2); // 2 = Header + Contact
}

Let's use a Floating Action Button (or FAB) to add info rows to our RecyclerView. Here is the new activity_main.xml that includes our FAB.

<RelativeLayout

    android:layout_width="match_parent"
    android:layout_height="match_parent"
    >

        />
    <android.support.design.widget.FloatingActionButton
        android:layout_width="@dimen/min_touch_size"
        android:layout_height="@dimen/min_touch_size"

        android:layout_marginEnd="@dimen/activity_horizontal_margin"
        android:onClick="onFabClicked"
        android:src="@android:mipmap/sym_def_app_icon"
        />
</RelativeLayout>

Now that the view is in our layout, let's hook it up with our MainActivity and make a way to add data to the adapter.

public void onFabClicked(View v) {
    RecyclerView recycler = (RecyclerView) findViewById(R.id.recycler);
    if (recycler != null) {
        MainRecyclerAdapter adapter = (MainRecyclerAdapter) recycler.getAdapter();
        if (adapter != null) {
            adapter.addInfoString("Test Info");
        }
    }
}

Awesome! Now we have a way to add data to our RecyclerView. However, you may have noticed a problem. When the screen fills up with rows of information, we will have a row at the bottom of the screen that is covered up by our FAB. Oh no!

Don't worry, we just have to add another view type to our adapter to account for having a FAB at the bottom of the screen. Let's just use a Space view for our footer that matches the size of our FAB. This will allow every row to always be visible.

holder_footer.xml

<Space
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="@dimen/min_touch_size"
    android:layout_marginBottom="@dimen/activity_vertical_margin"
    android:layout_marginEnd="@dimen/activity_horizontal_margin"
    />

With that xml file defined, a class needs to be defined to hold this view type.

public class FooterViewHolder extends RecyclerView.ViewHolder {
    public FooterViewHolder(View itemView) {
        super(itemView);
    }
}

Now, we can create a new view type in our MainRecyclerAdapter to represent this footer view type. Next, let's update our getItemViewType() method to find which position corresponds to this new type.

private static final int FOOTER_TYPE = 3;
//...
@Override
public int getItemViewType(int position) {
    // Get the footer position (the last item in the list)
    int footerPosition = getItemCount() - 1;

    if (position == 0) {
        return HEADER_TYPE;
    } else if (position == 1) {
        return CONTACT_TYPE;
    } else if (position == footerPosition) {
        return FOOTER_TYPE;
    } else {
        return INFO_TYPE;
    }
}

We also need to update our onCreateViewHolder() method to create the new footer view type.

case FOOTER_TYPE: {
    holder = new FooterViewHolder(inflater.inflate(R.layout.holder_footer, parent, false));
    break;
}

Lastly, we have to account for the size of our MainRecyclerAdapter now that we have a new footer view type.

@Override
public int getItemCount() {
    return 3 + mInfoStrings.size(); // 3 = Header + Contact + Footer
}

Perfect.

To wrap up this demo, we need to set up the delete icons to actually delete that row. We will do this by creating a listener interface in our MainActivity that our adapter will implement. This interface will take in the position that is being clicked so that it can remove the correct data and update the view accordingly.

public interface OnViewHolderClickedListener {
    void onViewHolderClicked(int position);
}

When our MainRecyclerAdapter implements this listener, we again will make use of the built in remove animation that the RecyclerView offers to us. By using notifyItemRemoved(), our RecyclerView cleanly moves all necessary elements and removes the desired element.

public class MainRecyclerAdapter extends RecyclerView.Adapter implements OnViewHolderClickedListener {
    @Override
    public void onViewHolderClicked(int position) {
        // Delete the data
        mInfoStrings.remove(position - 2); // minus 2 for the header and contact
        notifyItemRemoved(position); // Update UI
    }
}

To hook this into to our InfoViewHolder, we will require this interface type as a parameter in the constructor so that it can report it's clicks to the adapter. Our InfoViewHolder will also act as the click listener for our delete button so that it can report the it's position to the interface listener when it needs to be deleted. 

public class InfoViewHolder extends RecyclerView.ViewHolder implements View.OnClickListener {
    public View mDeleteView;
    public TextView mInfoTextView;
    private WeakReference<OnViewHolderClickedListener> mClickListener;

    public InfoViewHolder(View itemView, OnViewHolderClickedListener listener) {
        super(itemView);
        //...
        mDeleteView.setOnClickListener(this);
        mClickListener = new WeakReference<>(listener);
    }

    @Override
    public void onClick(View v) {
        if (mClickListener.get() != null) {
            mClickListener.get().onViewHolderClicked(getAdapterPosition());
        }
    }
}

It is important to use a WeakReference to keep hold of the reference to the adapter, so that the reference is not retained longer than the adapter wants to be retained. This prevents retain cycles which can cause memory leaks in our application.

Then we update onCreateViewHolder() to pass the adapter as a listener when creating the InfoViewHolder.

holder = new InfoViewHolder(inflater.inflate(R.layout.holder_info, parent, false), this);

And voila, we now have a RecyclerView set up with an adapter where we can add and remove rows and takes into account the space used up by our FAB so that our bottom rows aren't covered up. By using an interface, we were able to report communication up from a view holder to it's parent adapter. Using the topics covered here, you should now be able to do just about anything you need with a RecyclerView.

We didn’t cover how to use a GridLayoutManager, which would make the RecyclerView behave similarly to a GridView. Instead of laying out views in a single file line with the LinearLayoutManager, the manager would layout views in columns to make a grid of view holders. Or, if you really want to get crazy, you could use a StaggeredGridLayoutManager, which will lay views out of varying height in columns to similar to what Pinterest does or the Google Keep app.

Most apps use this view to accomplish varying tasks from simple lists of one view type (like a messages application) to sectionable lists that have many different types. A good example of this is the search in Spotify or Youtube that list different channels, playlists, and single pieces of media in one scrolling list. Typically, the best use case for a RecyclerView over a Scroll View is when the list will be modified while someone is using it. If all the data is static and won't change, you'll probably be better off with a Scroll View.
